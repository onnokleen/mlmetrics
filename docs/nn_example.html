8 Empirical Illustration: Neural Networks for Volatility Forecasting

8.1 Overview

The previous chapters introduced feed-forward neural networks, recurrent neural networks, and LSTMs as separate model classes. This chapter uses them in one empirical forecasting workflow. The goal is not to prove that one architecture is universally best. The goal is to show how to compare architectures when the outcome is a dependent financial time series and the forecast-origin information set matters.

The illustration is based on daily realized-volatility measures for the SPDR S&P 500 ETF (SPY) constructed from WRDS TAQ intraday data. The TAQ-derived data are proprietary and are therefore not shipped with the public book. The public page shows the design, the local replication code, and shareable derived performance summaries, but not the underlying volatility observations.

The comparison mirrors the linked Keras notebook but makes the econometric design stricter. All three architectures forecast the same target, use the same 22-trading-day history, use chronological train-validation-test splits, and estimate preprocessing quantities only from the training sample.

8.2 Roadmap

  1. We define the realized-volatility target and the one-step-ahead forecast origin.
  2. We convert the notebook workflow into a reproducible local-data pipeline.
  3. We compare an FNN, a Simple RNN, and an LSTM using the same information set.
  4. We use time-ordered validation and Optuna tuning without random cross-validation.
  5. We interpret the derived forecast-performance summary and close with computational replication tasks.

8.3 Data and Forecast Target

Let \(RV_t\) denote daily realized variance for SPY on trading day \(t\), computed from cleaned intraday prices. The chapter works with

\[ z_t = \log(RV_t), \]

because realized variance is positive and highly right-skewed. The forecasting problem is

\[ \widehat z_{t+1\mid t} = f_\theta(z_t,z_{t-1},\ldots,z_{t-21}), \]

where the input window contains the previous 22 trading days, roughly one trading month.

The same information set is used for all three architectures. The FNN receives the 22 lags as a flattened feature vector. The Simple RNN and LSTM receive the same values as an ordered sequence. This keeps the empirical comparison focused on architecture rather than on unequal access to history.

Proprietary Data

The local replication code expects a file such as data/taq_spy/SPY_daily_measures.csv with a daily realized-variance column rv. That directory is intentionally excluded from git because the underlying TAQ data are licensed. Public readers can inspect the workflow but cannot reproduce the empirical numbers without their own licensed data.

Figure 8.1 shows the forecast-origin convention. The training, validation, and test blocks are chronological. Scaling constants and hyperparameters are learned before the test block is touched.

The original notebook used Keras/TensorFlow, Optuna, min-max scaling, and three architectures: FNN, Simple RNN, and LSTM. The local workflow below keeps the same broad structure but changes the target construction and validation design.

The important econometric changes are:

  • the target is \(\log(RV_t)\) rather than an unspecified scaled level
  • each architecture receives the same 22-day history
  • scaling is estimated on the training block only
  • validation is chronological rather than a random split
  • the final test block is touched only once

The chunk below is intentionally not evaluated when the public book renders. It requires proprietary local data and TensorFlow/Keras.

Show the code 
import os
os.environ["KERAS_BACKEND"] = "tensorflow"

import random
import time
import numpy as np
import pandas as pd
import optuna
import keras
from keras import layers
from sklearn.metrics import mean_absolute_error, root_mean_squared_error

random.seed(0)
np.random.seed(0)

path = "data/taq_spy/SPY_daily_measures.csv"
rv = (
    pd.read_csv(path, parse_dates=["date"])
    .sort_values("date")
    .loc[:, ["date", "rv"]]
    .dropna()
)
rv = rv.loc[rv["rv"] > 0].reset_index(drop=True)
rv["z"] = np.log(rv["rv"])

The next helper constructs the supervised sequence problem. For each valid origin \(t\), the model sees \((z_t,\ldots,z_{t-21})\) and forecasts \(z_{t+1}\).

Show the code 
window = 22

def make_windows(values, window):
    X, y, target_index = [], [], []
    for end in range(window - 1, len(values) - 1):
        X.append(values[end - window + 1 : end + 1])
        y.append(values[end + 1])
        target_index.append(end + 1)
    return np.asarray(X), np.asarray(y), np.asarray(target_index)

X_raw, y_raw, target_index = make_windows(rv["z"].to_numpy(), window)
dates = rv.loc[target_index, "date"].reset_index(drop=True)

n = len(y_raw)
train_end = int(0.64 * n)
valid_end = int(0.80 * n)

X_train_raw, y_train_raw = X_raw[:train_end], y_raw[:train_end]
X_valid_raw, y_valid_raw = X_raw[train_end:valid_end], y_raw[train_end:valid_end]
X_test_raw, y_test = X_raw[valid_end:], y_raw[valid_end:]
test_dates = dates.iloc[valid_end:]

center = y_train_raw.mean()
scale = y_train_raw.std()

def standardize(a):
    return (a - center) / scale

def unstandardize(a):
    return a * scale + center

X_train = standardize(X_train_raw)
X_valid = standardize(X_valid_raw)
X_test = standardize(X_test_raw)
y_train = standardize(y_train_raw)
y_valid = standardize(y_valid_raw)

8.4 Comparable Architectures

Figure 8.2 shows the comparison. The input information is identical; only the map from the 22-day history to the forecast differs.

The Keras definitions below follow the notebook style. The only difference is that the input shapes enforce the same 22-day information set across architectures.

Show the code 
def make_fnn(n_neurons, learning_rate):
    model = keras.Sequential(
        [
            keras.Input(shape=(window,)),
            layers.Dense(n_neurons, activation="softplus"),
            layers.Dense(1),
        ],
        name="FNN",
    )
    model.compile(
        loss=keras.losses.MeanSquaredError(),
        optimizer=keras.optimizers.Adam(learning_rate=learning_rate),
    )
    return model

def make_simple_rnn(n_neurons, learning_rate):
    model = keras.Sequential(
        [
            keras.Input(shape=(window, 1)),
            layers.SimpleRNN(n_neurons, activation="softplus", return_sequences=False),
            layers.Dense(1),
        ],
        name="SimpleRNN",
    )
    model.compile(
        loss=keras.losses.MeanSquaredError(),
        optimizer=keras.optimizers.Adam(learning_rate=learning_rate),
    )
    return model

def make_lstm(n_neurons, learning_rate):
    model = keras.Sequential(
        [
            keras.Input(shape=(window, 1)),
            layers.LSTM(n_neurons, activation="softplus", return_sequences=False),
            layers.Dense(1),
        ],
        name="LSTM",
    )
    model.compile(
        loss=keras.losses.MeanSquaredError(),
        optimizer=keras.optimizers.Adam(learning_rate=learning_rate),
    )
    return model

8.5 Validation and Tuning

The tuning objective uses the validation block directly. This is not nested evaluation; it is a compact empirical illustration. The test block remains untouched until the selected hyperparameters are fixed.

Show the code 
def as_sequence(X):
    return X[..., np.newaxis]

def tune_model(model_factory, X_tr, X_va, n_trials=20):
    def objective(trial):
        params = {
            "learning_rate": trial.suggest_float("learning_rate", 1e-4, 1e-3, log=True),
            "batch_size": trial.suggest_int("batch_size", 16, 64, step=4),
            "n_neurons": trial.suggest_int("n_neurons", 16, 96, step=4),
            "epochs": 30,
        }
        model = model_factory(params["n_neurons"], params["learning_rate"])
        history = model.fit(
            X_tr,
            y_train,
            validation_data=(X_va, y_valid),
            epochs=params["epochs"],
            batch_size=params["batch_size"],
            verbose=0,
            shuffle=False,
        )
        return min(history.history["val_loss"])

    study = optuna.create_study(
        direction="minimize",
        sampler=optuna.samplers.TPESampler(seed=1),
    )
    study.optimize(objective, n_trials=n_trials)
    return study.best_params

best_fnn = tune_model(make_fnn, X_train, X_valid)
best_rnn = tune_model(make_simple_rnn, as_sequence(X_train), as_sequence(X_valid))
best_lstm = tune_model(make_lstm, as_sequence(X_train), as_sequence(X_valid))

After tuning, the selected model is refit on the combined train-validation block and evaluated once on the test block.

Show the code 
def qlike(rv_true, rv_pred):
    rv_pred = np.maximum(rv_pred, 1e-12)
    return np.mean(rv_true / rv_pred - np.log(rv_true / rv_pred) - 1)

def fit_and_score(name, model_factory, params, X_trva, y_trva, X_te):
    model = model_factory(params["n_neurons"], params["learning_rate"])
    model.fit(
        X_trva,
        y_trva,
        epochs=100,
        batch_size=params["batch_size"],
        verbose=0,
        shuffle=False,
    )
    z_hat = unstandardize(model.predict(X_te).ravel())
    rv_hat = np.exp(z_hat)
    rv_true = np.exp(y_test)
    return {
        "model": name,
        "rmse_log_rv": root_mean_squared_error(y_test, z_hat),
        "mae_log_rv": mean_absolute_error(y_test, z_hat),
        "qlike": qlike(rv_true, rv_hat),
        "forecast": z_hat,
    }

X_train_valid = np.vstack([X_train, X_valid])
y_train_valid = np.concatenate([y_train, y_valid])

results = [
    fit_and_score("FNN", make_fnn, best_fnn, X_train_valid, y_train_valid, X_test),
    fit_and_score("Simple RNN", make_simple_rnn, best_rnn, as_sequence(X_train_valid), y_train_valid, as_sequence(X_test)),
    fit_and_score("LSTM", make_lstm, best_lstm, as_sequence(X_train_valid), y_train_valid, as_sequence(X_test)),
]

performance = pd.DataFrame(results).drop(columns=["forecast"])
performance

8.6 Results

The table below reports derived test-set output from the local SPY realized-variance file. The raw TAQ-derived observations are not shipped with the public chapter; the table records only aggregate forecast errors from the non-evaluated replication workflow above. The tuning run used 20 Optuna trials per architecture, with all scaling constants estimated from the training block only.

Model RMSE log RV MAE log RV QLIKE
FNN 0.6806 0.5247 0.3195
Simple RNN 0.6509 0.5132 0.2760
LSTM 0.6515 0.5096 0.2824

Figure 8.3 should be interpreted cautiously. The Simple RNN and LSTM slightly improve test RMSE relative to the FNN in this run, but the gains are small and the two recurrent architectures are nearly tied. The more important lesson is methodological: with dependent financial data, the architecture comparison is only credible after the forecast target, lag window, scaling, validation block, and tuning protocol have been fixed.

The following plotting pattern regenerates Figure 8.3 once results and performance have been produced locally.

8.7 Summary

Key Takeaways
  1. Empirical architecture comparisons must hold the forecast target, information set, preprocessing, validation design, and tuning budget fixed.
  2. In this volatility example, all three networks forecast \(z_{t+1}=\log(RV_{t+1})\) from the same 22-day history.
  3. The FNN sees the history as a flattened vector, while the Simple RNN and LSTM process it as an ordered sequence.
  4. Time-ordered validation is required because random splits would leak future volatility regimes into tuning.
  5. Small RMSE differences should be interpreted as predictive evidence for this design, not as structural evidence that one architecture captures an economic mechanism.
Common Pitfalls
  • Scaling the full volatility series before the train-validation-test split.
  • Giving the RNN or LSTM a longer history than the FNN and then attributing the gain to recurrence.
  • Using Keras’s validation_split mechanically when it does not match the desired chronological validation block.
  • Shuffling observations during training or validation for a time-series forecast.
  • Treating proprietary-data results as reproducible unless the data access requirements are stated clearly.

8.8 Computational Exercises

Replication Task 8.1: Update the Local SPY Results

Using licensed access to the local SPY realized-volatility file:

  1. Run the non-evaluated Keras/TensorFlow chunks in this chapter.
  2. Update the published performance table with the resulting performance object.
  3. Regenerate Figure 8.3 from the local results object.
  4. Check whether the ordering of FNN, Simple RNN, and LSTM changes when the target is log realized variance rather than the notebook’s scaled volatility target.
  5. Briefly report whether the recurrent architectures improve RMSE enough to justify their additional complexity.
Replication Task 8.2: Diagnose Validation Leakage

Modify the workflow in two deliberately incorrect ways:

  1. Estimate the scaling constants using the full sample instead of the training block only.
  2. Replace the chronological validation block with a random validation split.

Compare the resulting validation and test losses with the disciplined workflow. The expected diagnostic pattern is that the incorrect workflow can make validation performance look better without producing a reliable improvement on the final test block.
